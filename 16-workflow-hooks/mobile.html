Hooks & Automation

Automated Quality Gates for Every Tool Call

A terminal session recording that walks through configuring hooks to auto-format code after edits, block dangerous commands, and enforce project standards — all without manual intervention.

WORKFLOW HOOKS TERMINAL RECORDING
OVERVIEW

What We'll Build

We configure three hooks that fire automatically during Claude Code sessions — no manual steps required once set up.

Auto-Formatter

  • Fires on PostToolUse
  • Matches Edit and Write tools
  • Runs Prettier on the changed file

Command Guard

  • Fires on PreToolUse
  • Matches Bash tool
  • Blocks destructive commands

Commit Linter

  • Fires on PreToolUse
  • Matches Bash with git commit
  • Validates conventional commit format

Workflow Write settings.json → Start Claude Code → Edit files → Watch hooks fire automatically

STEP 1

Write the Hooks Configuration

Hooks live in .claude/settings.json (project-scoped) or ~/.claude/settings.json (global). We'll use project scope.

.claude/settings.json
{
"hooks": {
"PostToolUse": [
{
"matcher": "Edit|Write",
"hooks": [
{
"type": "command",
"command": "npx prettier --write
$(echo $TOOL_INPUT | jq -r '.file_path')"
}
]
}
]
}
}

How it works After every Edit or Write tool call, the hook extracts the file_path from the tool input JSON and runs Prettier on it. Exit code 0 means success — the session continues seamlessly.

STEP 2

Add a Command Guard

A PreToolUse hook that blocks dangerous shell commands before they execute. Exit code 2 blocks the tool call.

.claude/settings.json — adding PreToolUse
"PreToolUse": [
{
"matcher": "Bash",
"hooks": [
{
"type": "command",
"command": "bash -c '
CMD=$(echo $TOOL_INPUT | jq -r \".command\")
if echo \"$CMD\" | grep -qE
\"rm -rf|DROP TABLE|--force|--no-verify\";
then
echo \"Blocked: destructive command\"
exit 2
fi'"
}
]
},
{
"matcher": "Bash(git commit*)",
"hooks": [
{
"type": "command",
"command": "bash -c '
MSG=$(echo $TOOL_INPUT | jq -r \".command\"
| grep -oP \"(?<=-m \\\").*?(?=\\\")\")
if ! echo \"$MSG\" | grep -qE
\"^(feat|fix|docs|style|refactor|test|chore):\";
then
echo \"Commit msg must use conventional format\"
exit 2
fi'"
}
]
}
]

Exit Code 2 = Block

The tool call is prevented and the hook's stderr/stdout is shown to Claude, which can adjust its approach

Matcher Patterns

Bash(git commit*) matches only Bash calls whose command starts with git commit — more precise than matching all Bash calls

STEP 3

Auto-Formatter in Action

We start Claude Code and ask it to edit a file. The PostToolUse hook fires automatically after each edit.

~/my-project — claude
> Add a calculateDiscount function to src/pricing.js
that accepts price and percentage parameters

│ Editing src/pricing.js...

✓ Edit applied to src/pricing.js

❯ Hook [PostToolUse → Edit]: running prettier...
❯ src/pricing.js 42ms

Added calculateDiscount(price, percentage) to src/pricing.js.
The function validates inputs and returns the discounted price.

Seamless The hook ran Prettier between the edit and Claude's response. The file on disk is always formatted — no manual step needed. Claude sees the hook ran successfully and continues normally.

STEP 4

Command Guard Blocks a Dangerous Command

We ask Claude to clean up some files. It attempts rm -rf and the hook blocks it.

~/my-project — claude
> Remove all files in the old-build directory

│ Preparing: rm -rf old-build/

✖ Hook [PreToolUse → Bash]: Blocked: destructive command
The command "rm -rf old-build/" was blocked by a project hook.

The project has a safety hook that prevents rm -rf commands.
I'll remove the files individually instead:

│ Running: ls old-build/
│ Hook [PreToolUse → Bash]: passed
│ Running: rm old-build/bundle.js old-build/styles.css
│ Hook [PreToolUse → Bash]: passed
│ Running: rmdir old-build
│ Hook [PreToolUse → Bash]: passed

✓ Removed 2 files and the old-build directory.

Self-correction Claude received the block message, understood the constraint, and found a safer alternative automatically. This is the power of hooks — they shape the agent's behaviour in real time.

STEP 5

Commit Message Linter

Claude tries to commit with a non-conventional message. The hook rejects it, and Claude reformats automatically.

~/my-project — claude
> Commit the pricing changes

│ Running: git add src/pricing.js
│ passed
│ Running: git commit -m "Add calculateDiscount function"

✖ Hook [PreToolUse → Bash(git commit*)]: Commit msg must use conventional format

The project requires conventional commit messages. Retrying:

│ Running: git commit -m "feat: add calculateDiscount function to pricing module"
│ Hook [PreToolUse → Bash(git commit*)]: passed
│ [main a3f8c2d] feat: add calculateDiscount function to pricing module

✓ Committed with conventional format: feat: add calculateDiscount...

Pattern Block → explain → retry. Hooks teach Claude your project's rules. After being blocked once, Claude tends to use the correct format for the rest of the session.

STEP 6

Inspect Active Hooks

The /hooks slash command shows all configured hooks grouped by event.

~/my-project — claude
> /hooks

Active Hooks (from .claude/settings.json)

PostToolUse
├ Edit|Write → npx prettier --write <file_path>

PreToolUse
├ Bash → Block rm -rf, DROP TABLE, --force, --no-verify
└ Bash(git commit*) → Enforce conventional commit format

3 hooks active across 2 events

Tip Use .claude/settings.local.json for personal hooks that shouldn't be committed to the repo. Both files are merged at runtime.

REFERENCE

Hook Events

EventFires WhenCan Block?
PreToolUseBefore any tool executesYes (exit 2)
PostToolUseAfter a tool completes successfullyNo
PostToolUseFailureAfter a tool call failsNo
UserPromptSubmitWhen the user sends a messageYes (exit 2)
SessionStartWhen a Claude Code session beginsNo
SessionEndWhen a session endsNo
SubagentStartWhen a sub-agent is launchedYes (exit 2)
NotificationWhen Claude sends a notificationNo
PreCompact / PostCompactBefore/after context compactionNo
StopWhen Claude finishes a response turnNo

Hook Types

  • command — run a shell command
  • prompt — single LLM call
  • agent — sub-agent verification
  • http — POST to an endpoint

Environment Variables

  • $TOOL_INPUT — JSON of tool parameters
  • $TOOL_OUTPUT — JSON of tool result (post only)
  • $SESSION_ID — current session ID
  • $CWD — current working directory

MCP Servers

Extend Claude Code with External Tool Servers

A terminal session recording that walks through adding MCP (Model Context Protocol) servers to Claude Code — giving it access to GitHub, databases, and custom APIs as native tools.

WORKFLOW MCP TERMINAL RECORDING
OVERVIEW

What We'll Build

We add two MCP servers — one via the CLI and one via a config file — then use them in a real workflow.

GitHub MCP Server

  • Added via claude mcp add CLI
  • Stored in ~/.claude.json (user-scoped)
  • Provides: create issues, list PRs, read repos

PostgreSQL MCP Server

  • Added via .mcp.json config file
  • Project-scoped — checked into version control
  • Provides: query, list tables, describe schema

Workflow Add servers → Verify with /mcp → Use tools in conversation → Combine MCP tools with built-in tools

STEP 1

Add GitHub MCP Server via CLI

The claude mcp add command registers a new server. The stdio transport spawns a local process.

~/my-project
$ claude mcp add stdio github \
npx @modelcontextprotocol/server-github \
--env GITHUB_TOKEN=ghp_xxxxxxxxxxxx

✓ Added MCP server github (stdio)
Command: npx @modelcontextprotocol/server-github
Scope: user (~/.claude.json)

$ claude mcp list

MCP Servers
● github (stdio) — user-scoped
npx @modelcontextprotocol/server-github

Scope User-scoped servers (stored in ~/.claude.json) are available in every project. Use --scope project to write to .mcp.json instead.

STEP 2

Add PostgreSQL MCP via Config File

For project-specific servers, write a .mcp.json file at the project root. This can be version-controlled.

.mcp.json
{
"mcpServers": {
"postgres": {
"command": "npx",
"args": [
"@modelcontextprotocol/server-postgres",
"postgresql://localhost:5432/myapp"
],
"env": {
"PGPASSWORD": "dev-password"
}
}
}
}

Config Format

command + args defines the process to spawn. env passes environment variables securely.

Version Control

Check in .mcp.json so every team member gets the same MCP servers. Use env vars for secrets.

STEP 3

Verify Servers in Session

Start Claude Code and use /mcp to see connected servers and their available tools.

~/my-project — claude
$ claude

╭──────────────────────────────────────────────────╮
│ Claude Code v1.0.32 │
│ 2 MCP servers connected │
╰──────────────────────────────────────────────────╯

> /mcp

MCP Servers

● github (stdio • user-scoped)
Tools:
mcp__github__create_issue
mcp__github__list_pull_requests
mcp__github__get_file_contents
mcp__github__search_repositories
... 12 more

● postgres (stdio • project-scoped)
Tools:
mcp__postgres__query
mcp__postgres__list_tables
mcp__postgres__describe_table

Naming convention MCP tools appear as mcp__<server>__<tool>. They work exactly like built-in tools — Claude can call them automatically based on your request.

STEP 4

Use GitHub Tools

Ask Claude Code about your GitHub repo. It automatically uses the MCP tools to fetch live data.

~/my-project — claude
> List open PRs on our repo and summarise each one

│ Calling mcp__github__list_pull_requests...
│ owner: "acme-corp", repo: "my-project", state: "open"

Open Pull Requests (3)

Seamless Claude decided which MCP tool to call, constructed the parameters, and presented the results — all from a natural language request.

STEP 5

Combine MCP + Built-in Tools

The real power: Claude chains MCP tools with built-in tools in a single workflow — read from the database, then edit code.

~/my-project — claude
> Check the users table schema, then update the
User TypeScript interface to match the actual columns

│ Calling mcp__postgres__describe_table...
│ table: "users"

│ Reading src/types/User.ts...

│ Editing src/types/User.ts...

✓ Updated src/types/User.ts to match the database schema:

+ avatar_url: string | null (new column in DB, was missing)
+ last_login_at: Date | null (new column in DB, was missing)
- phone: string (column dropped from DB)

Multi-tool chain One prompt triggered three tool calls across two systems: MCP (database) → built-in (read) → built-in (edit). Claude orchestrated the entire flow.

STEP 6

Create a GitHub Issue from Analysis

Claude finds a problem in the code and creates a GitHub issue directly — without leaving the terminal.

~/my-project — claude
> Check if there are any TODO comments in the codebase
and create GitHub issues for the important ones

│ Grepping for TODO across project...
│ Found 4 TODOs in 3 files

│ Calling mcp__github__create_issue...
│ title: "Handle pagination in /api/users endpoint"
│ body: "src/routes/users.js:45 has a TODO..."
│ labels: ["enhancement"]

│ Calling mcp__github__create_issue...
│ title: "Add retry logic for payment webhook"
│ labels: ["bug", "high-priority"]

✓ Created 2 GitHub issues:
#143 Handle pagination in /api/users endpoint
#144 Add retry logic for payment webhook
(Skipped 2 minor TODOs: formatting nits)

Judgment Claude didn't blindly create issues for every TODO — it assessed severity and only filed the ones that mattered. The MCP server is a tool, not a script.

REFERENCE

MCP Management Commands

CommandPurpose
claude mcp add stdio <name> <cmd> [args]Add a new stdio MCP server
claude mcp add sse <name> <url>Add a remote SSE (HTTP) MCP server
claude mcp listList all configured servers and their scope
claude mcp remove <name>Remove a server
/mcp (in session)View connected servers and their tools

Config File Locations

  • ~/.claude.json — user-scoped (all projects)
  • .mcp.json — project-scoped (version control)

Transport Types

  • stdio — local process, spawned by Claude Code
  • sse — HTTP Server-Sent Events (remote)

Tip Popular MCP servers: @modelcontextprotocol/server-github, server-postgres, server-filesystem, server-slack, server-google-drive. Browse the registry at mcp.so.

Claude Code Desktop

The Code Tab in the Claude Desktop App

A graphical interface for Claude Code — visual diff review, parallel Git-worktree sessions, app previews, PR monitoring, computer use, and scheduled tasks. Same engine as the CLI.

Code Tab macOS & Windows Visual Diffs Same Engine as CLI

Companion to presentation 11 (Claude Desktop overview); see also 13 (Cowork) and 14 (Chat tab). Based on code.claude.com/docs/en/desktop-quickstart and code.claude.com/docs/en/desktop.

01

What Is Claude Code Desktop?

“Claude Code Desktop” is the Code tab inside the Claude Desktop app. The app has three tabs:

Chat — deck 14

  • General conversation, Projects, Artifacts
  • No file-write access
  • Same backend as claude.ai
  • Original MCP surface (claude_desktop_config.json)

Code « this deck

  • Interactive coding assistant
  • Direct access to local files
  • You review and approve each change
  • Same engine as the CLI

Cowork — deck 13

  • Autonomous background agent
  • Runs in a cloud VM
  • Works independently while you do other work
  • Hosts Dispatch (task from your phone)

What Desktop adds on top of the standard Claude Code experience: visual diff review with inline comments, live app preview, computer use, GitHub PR monitoring with auto-fix and auto-merge, parallel sessions with automatic Git worktree isolation, Dispatch integration, scheduled tasks, connectors for external tools, and Local / SSH / Cloud environments.

Subscription

Claude Code in the desktop app requires a Pro, Max, Team, or Enterprise plan. Clicking the Code tab on a free plan prompts you to upgrade.

First Launch

  1. Run the installer
  2. Launch from Applications / Start menu
  3. Sign in with your Anthropic account
  4. Click the Code tab at the top center
  5. If you get a 403 or auth error, sign out and back in from the app menu

Cowork Tab Caveat

The Cowork tab requires Apple Silicon (M1 or later) on macOS. On Intel Macs, the Chat and Code tabs still work normally. On Windows, Cowork works on all supported hardware.

CLI Is Separate

The desktop app includes Claude Code internally — you don’t need Node.js or the CLI for the Code tab. If you want to use claude from the terminal as well, install the CLI separately.

03

Starting a Session

Before sending your first message, configure four things in the prompt area:

1. Environment

  • Local — runs on your machine, direct file access
  • Remote — Anthropic cloud, continues if you close the app
  • SSH — a remote machine you manage (Claude Code must be installed on it)

2. Project Folder

  • Click Select folder and pick your project directory
  • Remote sessions support multiple repositories via the + button next to the repo pill

3. Model

  • Dropdown next to the send button
  • Opus, Sonnet, Haiku — see docs for comparison
  • Locked once the session starts

4. Permission Mode

  • Mode selector next to the send button
  • Can be changed during the session
  • Defaults to Ask permissions for new users

Sample tasks: Find a TODO comment and fix it · Add tests for the main function · Create a CLAUDE.md with instructions for this codebase. You can interrupt Claude any time with the stop button or by typing a correction and pressing Enter.

04

Permission Modes

ModeSettings keyBehavior
Ask permissions default Asks before editing files or running commands. You see a diff and accept or reject each change. Recommended for new users.
Auto accept edits acceptEdits Auto-accepts file edits and common filesystem commands (mkdir, touch, mv), still asks before other terminal commands.
Plan mode plan Reads files and runs commands to explore, then proposes a plan without editing source code. Good before a large refactor.
Auto auto Executes all actions with background safety checks. Research preview. Team, Enterprise, and API plans only. Requires Sonnet 4.6 or Opus 4.6. Enable in Settings → Claude Code.
Bypass permissions bypassPermissions No permission prompts. Equivalent to --dangerously-skip-permissions. Off by default; enable under Settings → Claude Code → “Allow bypass permissions mode”. Enterprise admins can disable this option.

Best Practice

Start complex tasks in Plan mode so Claude maps out an approach first. Approve the plan, then switch to Auto accept edits to execute it quickly.

Remote Session Differences

Remote sessions support Auto accept edits and Plan mode only. Ask permissions is not available (remote auto-accepts edits) and Bypass permissions is not available (the remote environment is already sandboxed). The dontAsk mode is CLI-only.

05

Diff View, Code Review, and Adding Context

Review Changes with Diff View

  • After Claude changes files, a diff stats indicator appears: +12 -1
  • Click it to open the diff viewer — file list on the left, changes on the right
  • Click any line in the diff to open a comment box; press Enter to add
  • Submit all comments at once: Cmd+Enter on macOS, Ctrl+Enter on Windows
  • Claude reads the comments and revises; updates appear as a new diff

Review Code Button

  • In the diff view, top-right toolbar
  • Asks Claude to evaluate current diffs and leave inline suggestions
  • Focuses on compile errors, logic errors, security issues, obvious bugs
  • Does not flag style, formatting, or lint-level issues

Add Files and Context

  • @mention files: type @ then a filename with autocomplete. Not available in remote sessions.
  • Attach files: images, PDFs, etc. via the attachment button
  • Drag and drop files directly into the prompt

Rejection Flow

If you reject a change, Claude asks how you’d like to proceed differently. Your files aren’t modified until you accept.

Context Compaction

When context fills up, Claude automatically summarizes the conversation and continues working. Type /compact to trigger summarization earlier and free up context space.

06

Parallel Sessions & Git Worktrees

Click + New session in the sidebar to work on multiple tasks in parallel. Each session tracks its own context and changes independently.

Automatic Git Worktrees

  • For Git repos, each session gets its own isolated copy via Git worktrees
  • Changes in one session don’t affect others until you commit
  • Default location: <project-root>/.claude/worktrees/
  • Custom location: Settings → Claude Code → “Worktree location”
  • Set a branch prefix to keep Claude-created branches organized
  • Archive a worktree from the sidebar: hover the session, click the archive icon

Including Ignored Files

Create a .worktreeinclude file in your project root to copy gitignored files like .env into new worktrees.

Sidebar Controls

  • Filter by status (Active, Archived)
  • Filter by environment (Local, Cloud)
  • Rename a session — click its title in the toolbar
  • Check context usage — click the session title

Git Is Required

Session isolation requires Git. Most Macs include it. On Windows, Git is required for the Code tab: install Git for Windows, then restart the app.

Continue in Another Surface

From the VS Code icon in the bottom right of the session toolbar:

  • Claude Code on the Web — pushes your branch, generates a conversation summary, creates a remote session with full context. Requires a clean working tree. Not available for SSH sessions.
  • Your IDE — opens your project in a supported IDE at the current working directory
07

Environments: Local, Remote, SSH

Local

  • Runs on your machine with direct file access
  • Inherits PATH and a fixed set of Claude Code variables from your shell profile on macOS
  • Other shell exports are not picked up — set them in the local environment editor (gear icon on Local in the environment dropdown)
  • Variables are encrypted on disk and apply to all local sessions and preview servers

Remote (Cloud)

  • Runs on Anthropic's cloud infrastructure
  • Continues in the background even if you close the app
  • Usage counts toward your subscription plan limits — no separate compute charges
  • Monitor from claude.ai/code or the Claude iOS app
  • Supports multiple repositories in one session
  • Custom cloud environments with network-access levels

SSH

  • Connect to a remote machine over SSH
  • Environment dropdown → + Add SSH connection
  • Fields: Name, SSH Host, SSH Port (default 22), Identity File
  • Claude Code must be installed on the remote machine
  • Supports permission modes, connectors, plugins, and MCP servers

Extended Thinking Tuning

Extended thinking is on by default. To disable entirely, set MAX_THINKING_TOKENS=0 in the local environment editor. On Opus 4.6 and Sonnet 4.6, other MAX_THINKING_TOKENS values are ignored because adaptive reasoning controls thinking depth — to force a fixed budget, also set CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1.

08

App Preview & Auto-Verify

Claude can start a dev server and open an embedded browser to verify its changes. Works for frontend apps and backend servers: Claude can test API endpoints, view server logs, and iterate on what it finds.

What You Can Do from the Preview Panel

  • Interact with your running app in the embedded browser
  • Watch Claude verify its own changes — screenshots, DOM inspection, clicking, filling forms, fixing issues
  • Start or stop servers from the Preview dropdown
  • Persist cookies and local storage across server restarts with Persist sessions
  • Edit server configuration or stop all servers at once

Configuration

Claude stores dev server config in .claude/launch.json at the root of the selected folder. JSON with comments. Edit directly or via Edit configuration in the Preview dropdown.

{
  "version": "0.0.1",
  "configurations": [
    {
      "name": "my-app",
      "runtimeExecutable": "npm",
      "runtimeArgs": ["run", "dev"],
      "port": 3000
    }
  ]
}

Auto-Verify

  • On by default
  • Claude automatically verifies changes after every edit
  • Screenshots, error checks, confirmation that changes work
  • Disable per project with "autoVerify": false in .claude/launch.json, or toggle from the Preview dropdown

Config Fields

  • runtimeExecutable + runtimeArgs — run via a package manager
  • program + args — standalone Node.js script
  • port (default 3000), cwd, env
  • autoPort — find a free port if the chosen one is taken

Multiple Servers

Define multiple entries in configurations for monorepos — e.g. a frontend with autoPort: true and an API server that requires a specific port.

09

Pull Request Monitoring

After you open a pull request, a CI status bar appears in the session. Claude Code uses the GitHub CLI (gh) to poll check results and surface failures.

Auto-Fix

  • Toggle in the CI status bar
  • Claude reads the failure output and iterates to fix failing CI checks

Auto-Merge

  • Toggle in the CI status bar
  • Merges the PR once all checks pass
  • Merge method is squash
  • Must also be enabled in your GitHub repository settings

Desktop Notification

Claude Code sends a desktop notification when CI finishes.

Prerequisite: gh CLI

PR monitoring requires the GitHub CLI (gh) to be installed and authenticated on your machine. If it’s not installed, Desktop prompts you to install it the first time you try to create a PR.

Scheduled Tasks

Separately, the desktop app supports scheduled tasks that run Claude on a recurring basis — for example a daily code review, a weekly dependency audit, or a briefing pulled from connected tools. Configured from the Code tab; see desktop-scheduled-tasks in the docs.

Dispatch Integration

Tasks sent to Dispatch (in the Cowork tab) from your phone can spawn Code sessions here. These appear in the sidebar with a Dispatch badge. Dispatch requires a Pro or Max plan and is not available on Team or Enterprise.

10

Computer Use

Computer use lets Claude open your apps, control your screen, and work on your machine the way you would — testing a native app in a mobile simulator, interacting with a desktop tool that has no CLI, automating something only available through a GUI.

Availability

  • Research preview on macOS and Windows
  • Pro or Max plan only
  • Not available on Team or Enterprise plans
  • Off by default — enable in Settings → General → Computer use

macOS Permission Grants

  • Accessibility — lets Claude click, type, and scroll
  • Screen Recording — lets Claude see what’s on your screen

How Claude Chooses a Tool

  1. If a connector exists for the service, use it
  2. If the task is a shell command, use Bash
  3. If it’s browser work and Claude in Chrome is set up, use that
  4. Otherwise fall back to computer use

Computer use is the broadest and slowest option, reserved for things no other tool can reach.

Per-App Access Tiers

TierWhat Claude can doApplies to
View onlySee the app in screenshotsBrowsers, trading platforms
Click onlyClick and scroll, no typingTerminals, IDEs
Full controlClick, type, drag, shortcutsEverything else

Approval Flow

The first time Claude needs an app, a prompt appears — Allow for this session or Deny. Approvals last the session, or 30 minutes in Dispatch-spawned sessions. Denied apps can be pre-listed in Settings.

Connectors, Plugins, Skills, and MCP

Connectors

  • Click + next to the prompt box → Connectors
  • Integrations for Google Calendar, Slack, GitHub, Linear, Notion, and more
  • Graphical setup flow — connectors are MCP servers under the hood
  • Manage via Settings → Connectors
  • Not available in remote sessions (the + button is hidden there)

Plugins

  • + button → Plugins — see installed plugins and their skills
  • Plugin browser for configured marketplaces including the official Anthropic marketplace
  • Enable, disable, or uninstall via Manage plugins
  • Scoped to user, project, or local-only
  • Not available for remote sessions

Skills / Slash Commands

  • Type / in the prompt box, or click + → Slash commands
  • Built-in commands, custom skills, project skills, plugin skills
  • Claude loads skills automatically when relevant
  • Or invoke one directly: the skill appears highlighted in the input field and you type your task after it

MCP Servers

  • Code tab is a full MCP host for local and SSH sessions
  • Supports stdio (local subprocess) and Streamable HTTP (HTTP or HTTPS)
  • Configure via Connectors UI, ~/.claude.json, or project .mcp.json
  • Add from CLI: claude mcp add <name> <cmd> (stdio) or claude mcp add --transport http <name> <url>
  • The Code tab connects from your machine — plain HTTP and private-network IPs work (e.g. http://10.60.12.34:3001/mcp)
  • Chat-tab MCP config (claude_desktop_config.json) is separate and does not appear here

Hooks

Hooks defined in settings apply to both the CLI and Desktop. No separate desktop-only hook system.

12

Coming from the CLI: Shared Config & Comparison

Desktop and CLI run the same underlying engine. On macOS, they read the same config files — MCP servers added via the CLI appear in the Desktop Code tab automatically. On Windows, the standard MSIX installer sandboxes Desktop’s filesystem, so config is not shared.

macOS — Config Is Shared

  • CLAUDE.md and CLAUDE.local.md in your project
  • MCP servers in ~/.claude.json or .mcp.json
  • Hooks and skills defined in settings
  • Settings in ~/.claude.json and ~/.claude/settings.json
  • Add an MCP server via claude mcp add and it appears in both CLI and Desktop Code tab

Windows (MSIX) — Config Is Not Shared

  • The standard Windows installer is an MSIX package that sandboxes Desktop’s filesystem
  • CLI writes to C:\Users\user\.claude.json (real filesystem)
  • Desktop reads from %LOCALAPPDATA%\Packages\Claude_…\LocalCache\ (virtualized)
  • MCP servers added via claude mcp add in PowerShell do not appear in the Desktop Code tab
  • Add MCP servers through the Desktop’s own UI (Connectors / + button) instead
  • WSL2 adds a third separate config surface (/home/user/.claude.json)

Move a Session into Desktop

Run /desktop in the terminal. Claude saves your session and opens it in the desktop app, then exits the CLI. macOS and Windows only.

Enterprise Configuration

Admin Console Controls

  • Code in the desktop — control whether users can access Claude Code in the desktop app
  • Code in the web — enable or disable web sessions
  • Remote Control — enable or disable Remote Control
  • Disable Bypass permissions mode — prevent users from enabling it

Managed Settings (Selected Keys)

KeyPurpose
permissions.disableBypassPermissionsModePrevent Bypass mode
disableAutoModeRemove Auto from the mode selector
autoModeCustomize the Auto-mode classifier rules

Managed settings override project and user settings and apply when Desktop spawns CLI sessions. Remote managed settings pushed through the admin console currently apply to CLI and IDE sessions only; for Desktop-specific restrictions use the admin console controls above.

Device Management

  • macOS — com.anthropic.Claude preference domain via Jamf, Kandji, etc.
  • Windows — registry at SOFTWARE\Policies\Claude
  • Policies include enabling/disabling Claude Code, controlling auto-updates, setting a custom deployment URL

Authentication & SSO

Enterprise organizations can require SSO for all users. SAML and OIDC supported — see the Anthropic Setting up SSO support article.

Deployment

  • macOS — distribute the .dmg via MDM (Jamf, Kandji)
  • Windows — deploy via MSIX package or .exe installer, including silent installation

Data Handling

Local sessions process code locally; remote sessions process it on Anthropic’s cloud. Conversations and code context are sent to Anthropic’s API for processing regardless.

What’s Not in Desktop & Troubleshooting

Not Available in Desktop

  • Linux — macOS and Windows only
  • Third-party providers — Bedrock, Vertex, Foundry (use the CLI)
  • Inline code suggestions — Desktop works through conversational prompts, not autocomplete
  • Agent teams — multi-agent orchestration is CLI / Agent SDK
  • Scripting and automation — no --print, no Agent SDK headless mode
  • --allowedTools / --disallowedTools — no equivalent
  • --verbose — check Console.app (macOS) or Event Viewer (Windows) instead
  • dontAsk permission mode — CLI only

Common Fixes

  • 403 / auth errors → sign out and back in from the app menu
  • Blank or stuck screen → restart the app (it auto-updates on launch)
  • “Failed to load session” → folder missing, Git LFS not installed, or permissions
  • Can’t find npm/node → check shell profile PATH, restart the app
  • Windows “Git is required” → install Git for Windows, restart
  • MCP servers misbehaving on Windows → verify config, restart, check logs

When to Use Desktop vs CLI

From the docs: “Use Desktop when you want visual diff review, file attachments, or session management in a sidebar. Use the CLI when you need scripting, automation, third-party providers, or prefer a terminal workflow.”

Config Sharing: macOS Only

CLI and Desktop share CLAUDE.md, MCP config, and settings on macOS. On Windows, the MSIX installer sandboxes Desktop — add MCP servers through the Desktop UI, not the CLI. WSL2 adds a third separate config.

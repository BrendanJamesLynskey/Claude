Multi-Agent Parallel Work

Worktree Isolation & Concurrent Agent Execution

A terminal session recording that demonstrates launching multiple agents in parallel, each working in an isolated Git worktree, then merging their independent contributions.

WORKFLOW PARALLEL TERMINAL RECORDING
OVERVIEW

What We'll Build

We define three specialised agents and launch them simultaneously on different parts of a codebase. Each works in its own Git worktree — no merge conflicts during execution.

Key benefit Three agents run concurrently — what would take 15 minutes sequentially completes in 5 minutes. Each worktree is a full Git checkout.

STEP 1

Define Three Specialised Agents

Each agent has a focused role, its own tool set, and isolation: worktree for safe parallel execution.

api-migrator/AGENT.md
---
name: api-migrator
description: Migrates API
endpoints from v1 to v2
tools: Read, Grep, Glob,
Edit, Write, Bash
model: sonnet
isolation: worktree
---

Migrate all v1 API routes
to the v2 pattern. Update
imports, response shapes,
and error handling.
docs-generator/AGENT.md
---
name: docs-generator
description: Generates API
documentation from source
tools: Read, Grep, Glob,
Write
model: sonnet
isolation: worktree
---

Generate OpenAPI specs and
markdown docs for every
route handler. Include
request/response examples.
test-coverage/AGENT.md
---
name: test-coverage
description: Increases test
coverage for untested files
tools: Read, Grep, Glob,
Edit, Write, Bash
model: sonnet
isolation: worktree
maxTurns: 40
---

Find files with <80% test
coverage. Write tests and
verify they pass.

Note The docs-generator has no Edit or Bash — it can only create new files with Write. Principle of least privilege.

STEP 2

Launch All Three in Parallel

We ask Claude Code to tackle all three tasks at once. It recognises each maps to a different agent and launches them concurrently.

~/my-project — claude
> I need three things done in parallel:
1. Migrate all v1 API routes to v2
2. Generate API docs for every endpoint
3. Add tests for any files below 80% coverage

▶ Launching 3 agents in parallel

▶ api-migrator → worktree /tmp/claude-wt-m1a2b3
▶ docs-generator → worktree /tmp/claude-wt-d4e5f6
▶ test-coverage → worktree /tmp/claude-wt-t7g8h9

All agents running concurrently...

Parallel execution Each agent gets its own Git worktree — a separate directory with a full checkout of the repo. They can edit the same files without conflicting during execution.

STEP 5

Merge All Branches

We ask Claude to merge all three branches. Because they touched different files, the merges are clean.

GUIDANCE

When to Use Parallel Agents

Good Candidates

  • Tasks that touch different files
  • Independent concerns: tests, docs, migration
  • Large refactors across many modules
  • Multi-language projects (backend + frontend)
  • Generating boilerplate across services

Poor Candidates

  • Tasks with sequential dependencies
  • Changes to a single file from multiple agents
  • Work that requires shared state or context
  • Small changes faster done in one session

Worktree Lifecycle

  • Created when agent starts
  • Preserved if agent made changes (branch returned)
  • Cleaned up if no changes were made

Conflict Strategy

  • Design agents to touch different file sets
  • Use the main session to merge and resolve
  • Run tests after merging to catch integration issues
SUMMARY

What We Covered

Parallel Launch

Multiple agents launched simultaneously from a single prompt — each in its own worktree

Worktree Isolation

Full Git checkouts ensure agents can't interfere with each other during execution

Branch Preservation

Each agent's changes land on a named branch for independent review or bulk merge

Conflict Resolution

The main session merges branches and resolves conflicts with full context from both sides

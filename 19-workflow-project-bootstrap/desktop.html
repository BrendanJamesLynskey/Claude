STEP 1

Bootstrap with /init

The /init command scans your project and generates a CLAUDE.md with architecture notes, conventions, and build commands.

~/my-project — claude
> /init

Scanning project structure...
│ Detected: Node.js + TypeScript + React
│ Package manager: pnpm
│ Test framework: Vitest
│ Linter: ESLint + Prettier
│ CI: GitHub Actions

✓ Created CLAUDE.md with project context

Sections generated:
• Project overview and architecture
• Build & run commands (dev, test, lint, build)
• Code conventions detected from existing code
• Directory structure summary
• Key dependencies and their purposes

What /init detects Language, framework, package manager, test runner, linter, CI system, directory layout, key dependencies. It reads package.json, config files, and samples source files.

STEP 2

Review the Generated CLAUDE.md

The generated file gives Claude Code project context that persists across every session.

CLAUDE.md (generated)
1 # Project: my-project
2
3 ## Architecture
4 React 18 SPA with Express API backend.
5 Monorepo: packages/web (frontend), packages/api (backend)
6
7 ## Commands
8 - Dev server: `pnpm dev`
9 - Run tests: `pnpm test`
10 - Lint: `pnpm lint`
11 - Build: `pnpm build`
12 - Type check: `pnpm typecheck`
13
14 ## Conventions
15 - TypeScript strict mode
16 - Functional React components with hooks
17 - CSS Modules for styling
18 - Barrel exports (index.ts) per directory
19
20 ## Directory Structure
21 packages/web/src/components/ - React components
22 packages/web/src/hooks/ - Custom React hooks
23 packages/api/src/routes/ - Express route handlers
24 packages/api/src/middleware/ - Express middleware

Loaded automatically CLAUDE.md is injected into every Claude Code session. Add project-specific rules here — e.g., "always use parameterised queries", "never import from internal packages directly".

STEP 3

Create a /deploy Skill

Skills are reusable slash commands defined as Markdown files in .claude/skills/.

~/my-project — claude
> Create a /deploy skill that deploys to staging

│ Creating .claude/skills/deploy/SKILL.md...

✓ Created skill /deploy
.claude/skills/deploy/SKILL.md
1 ---
2 name: deploy
3 description: Deploy the app to staging environment
4 allowed-tools: Bash, Read
5 ---
6
7 Deploy the application to the staging environment:
8
9 1. Run the full test suite: `pnpm test`
10 2. Run the type checker: `pnpm typecheck`
11 3. Build the project: `pnpm build`
12 4. Deploy: `pnpm deploy:staging`
13 5. Verify the health endpoint responds
14
15 If any step fails, stop and report the error.
16 Do not proceed to the next step on failure.

allowed-tools restricts what tools the skill can use. This deploy skill gets Bash and Read only — it can run commands and read files but can't edit source code.

STEP 4

Create a /component Skill with Arguments

Skills can accept arguments via $ARGUMENTS. This one scaffolds a React component with tests and styles.

.claude/skills/component/SKILL.md
1 ---
2 name: component
3 description: Scaffold a new React component
4 argument-hint: ComponentName
5 ---
6
7 Create a new React component named $ARGUMENTS:
8
9 1. Component file: `packages/web/src/components/$ARGUMENTS/$ARGUMENTS.tsx`
10 - Functional component with TypeScript props interface
11 - Use CSS Modules for styling
12 2. Styles: `packages/web/src/components/$ARGUMENTS/$ARGUMENTS.module.css`
13 3. Tests: `packages/web/src/components/$ARGUMENTS/$ARGUMENTS.test.tsx`
14 - Render test with React Testing Library
15 4. Barrel export: `packages/web/src/components/$ARGUMENTS/index.ts`
16
17 Follow the project's existing component patterns.
18 Check an existing component for reference before creating.

$ARGUMENTS is replaced with whatever follows the skill name. So /component UserProfile creates a UserProfile component with all four files.

STEP 5

Invoke the /component Skill

We use the skill to scaffold a new component. Claude reads an existing component first, then creates the new one to match.

~/my-project — claude
> /component UserProfile

│ Reading packages/web/src/components/Header/Header.tsx
│ (reference for project patterns)

│ Creating packages/web/src/components/UserProfile/
│ ✓ UserProfile.tsx (component + props interface)
│ ✓ UserProfile.module.css (CSS Module skeleton)
│ ✓ UserProfile.test.tsx (render + snapshot test)
│ ✓ index.ts (barrel export)

✓ Created UserProfile component (4 files)
Matches project patterns: TypeScript strict, CSS Modules, RTL tests

Pattern matching The skill told Claude to "check an existing component for reference" — so it read Header.tsx first and matched the exact style, import patterns, and test conventions already in the project.

STEP 6

Invoke the /deploy Skill

The deploy skill runs a multi-step pipeline — tests, type check, build, deploy, verify — stopping on first failure.

~/my-project — claude
> /deploy

Step 1/5: Running tests...
│ pnpm test
│ 87 tests passed (4.2s)

Step 2/5: Type checking...
│ pnpm typecheck
│ No errors found

Step 3/5: Building...
│ pnpm build
│ Build completed (12.8s)

Step 4/5: Deploying to staging...
│ pnpm deploy:staging
│ Deployed to staging-my-project.example.com

Step 5/5: Verifying health endpoint...
│ curl https://staging-my-project.example.com/health
│ {"status":"ok","version":"1.4.2"}

✓ Staging deploy complete
All 5 steps passed. App is live at staging-my-project.example.com

Repeatable Every team member gets the same deploy workflow with /deploy. No need to remember the sequence — the skill encodes your team's deployment procedure.

STEP 7

Project Memory with /memory

The /memory command lets you view and edit what Claude Code remembers about your project across sessions.

~/my-project — claude
> /memory

Project Memory (CLAUDE.md)

Showing current CLAUDE.md contents for editing.
This file is loaded into every session automatically.

──────────────────────────────────────────
# Project: my-project
## Architecture
React 18 SPA with Express API backend...
──────────────────────────────────────────

> Add to CLAUDE.md: "When writing database queries,
always use the query builder in packages/api/src/db/
instead of raw SQL. This is a security requirement."

│ Editing CLAUDE.md...
✓ Added security rule to CLAUDE.md under Conventions.
This will be enforced in all future sessions.

Persistence CLAUDE.md is just a file in your repo. Commit it to version control and every team member's Claude Code sessions inherit the same project knowledge, conventions, and rules.

REFERENCE

File Locations

FileLocationScope
Project memoryCLAUDE.md (project root)All sessions in this project
Personal memory~/.claude/CLAUDE.mdAll projects, your sessions only
Project skills.claude/skills/<name>/SKILL.mdAll team members in this project
Personal skills~/.claude/skills/<name>/SKILL.mdAll your projects
Project agents.claude/agents/<name>/AGENT.mdAll team members in this project
Project settings.claude/settings.jsonAll team members (hooks, etc.)
Local settings.claude/settings.local.jsonYour machine only (gitignored)

Tip Commit CLAUDE.md, .claude/skills/, .claude/agents/, and .claude/settings.json to version control. Keep .claude/settings.local.json gitignored for personal preferences.

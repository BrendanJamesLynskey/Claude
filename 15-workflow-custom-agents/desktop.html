Custom Agents

Build, Configure & Deploy Specialised Sub-Agents

A terminal session recording that walks through creating custom agent definitions, configuring their capabilities, and watching Claude Code delegate work to them automatically.

WORKFLOW AGENTS TERMINAL RECORDING
OVERVIEW

What We'll Build

In this walkthrough we create two custom agents — a code reviewer and a test writer — then trigger a task that causes Claude Code to delegate to them automatically.

Code Reviewer Agent

  • Runs in an isolated worktree
  • Uses Read, Grep, Glob tools only
  • Reviews code for bugs, security, and style
  • Returns structured findings

Test Writer Agent

  • Has access to Read, Edit, Write, Bash
  • Analyses existing code to generate tests
  • Runs the tests to confirm they pass
  • Uses Sonnet model for speed

Workflow Create agents → Write source code → Ask Claude to review & test → Watch delegation happen

STEP 1

Scaffold the Agent Directory

Agent definitions live in .claude/agents/. Each agent gets its own directory with an AGENT.md file.

~/my-project
$ mkdir -p .claude/agents/code-reviewer .claude/agents/test-writer
$ tree .claude/agents/
.claude/agents/
├── code-reviewer/
└── test-writer/

# Two directories ready — each will hold an AGENT.md

Note Agents can also live in ~/.claude/agents/ for personal agents available across all projects.

STEP 2

Define the Code Reviewer Agent

The YAML frontmatter controls the agent's capabilities. The Markdown body is its system prompt.

.claude/agents/code-reviewer/AGENT.md
1 ---
2 name: code-reviewer
3 description: Reviews code for bugs, security issues, and style
4 tools: Read, Grep, Glob
5 model: opus
6 isolation: worktree
7 ---
8
9 You are a senior code reviewer. Analyse the code
10 in this repository and report:
11
12 1. **Bugs** — logic errors, off-by-ones, null refs
13 2. **Security** — injection, auth bypasses, secrets
14 3. **Style** — naming, dead code, complexity
15
16 For each finding, cite the file and line number.
17 Rate severity: critical / warning / info.
18 Be concise. No false positives.

tools: Read, Grep, Glob

Read-only access — this agent cannot modify files

isolation: worktree

Runs in a separate Git worktree — no interference with your working tree

model: opus

Uses the most capable model for thorough review quality

STEP 3

Define the Test Writer Agent

This agent gets write access and Bash so it can create test files and run them.

.claude/agents/test-writer/AGENT.md
1 ---
2 name: test-writer
3 description: Generates and runs unit tests for source files
4 tools: Read, Grep, Glob, Edit, Write, Bash
5 model: sonnet
6 isolation: worktree
7 maxTurns: 30
8 ---
9
10 You are a test engineer. For each source file:
11
12 1. Read the implementation
13 2. Identify edge cases and key paths
14 3. Write comprehensive unit tests
15 4. Run the tests and fix any failures
16
17 Use the project's existing test framework.
18 Aim for >90% branch coverage.

model: sonnet

Sonnet is fast and cost-effective for test generation — the review agent uses Opus for deeper analysis

maxTurns: 30

Caps the agent at 30 tool calls — prevents runaway loops while allowing iterative test-fix cycles

STEP 4

Write Some Source Code to Review

We start a Claude Code session and ask it to create a small Express API with a deliberate bug for the agents to find.

~/my-project — claude
$ claude

╭──────────────────────────────────────────────────╮
│ Claude Code v1.0.32 │
│ /help for commands • /agents to manage agents │
╰──────────────────────────────────────────────────╯

> Create src/users.js with a simple Express router
that has GET /users/:id and POST /users endpoints.
Include a subtle SQL injection vulnerability in the
GET route for demonstration purposes.

✓ Created src/users.js with Express router
• GET /users/:id — fetches user by ID (vulnerable to SQLi)
• POST /users — creates a new user
• Uses raw SQL query string interpolation on the GET route

Why? We want realistic code for our agents to analyse. The SQL injection gives the code-reviewer something concrete to flag.

STEP 5

Trigger the Code Reviewer

We ask Claude Code to review the codebase. Because we have a code-reviewer agent whose description matches, Claude delegates the task.

~/my-project — claude
> Review this codebase for bugs and security issues

▶ Delegating to agent: code-reviewer
model: opus • isolation: worktree • tools: Read, Grep, Glob

Creating worktree at /tmp/claude-wt-a3f8c2...
Agent started • reviewing files...

│ Reading src/users.js...
│ Grepping for query patterns...
│ Checking route handlers...
│ Scanning for input validation...

✓ Agent complete (8 tool calls, 42s)

Key insight The delegation happened automatically — Claude matched the task description ("review...bugs and security") to the agent's description field. You can also force delegation with use the code-reviewer agent.

STEP 6

Review Findings

The code-reviewer agent returns its findings to the main session. The worktree is automatically cleaned up.

~/my-project — claude (code-reviewer results)
Code Review — 3 findings

✖ CRITICAL SQL Injection in src/users.js:14
The GET /users/:id route interpolates `req.params.id`
directly into a SQL string:
db.query(`SELECT * FROM users WHERE id = ${id}`)
Fix: use parameterised queries — db.query('SELECT * FROM users WHERE id = $1', [id])

⚠ WARNING Missing input validation in src/users.js:28
POST /users does not validate req.body.email format
before insertion. Could store malformed data.

ⓘ INFO No rate limiting on POST /users
Consider adding express-rate-limit to prevent abuse.

Worktree /tmp/claude-wt-a3f8c2 cleaned up (no changes made).

Worktree lifecycle Because the code-reviewer is read-only and made no file changes, the worktree was automatically removed. If it had made changes, the worktree path and branch would be returned for you to merge.

STEP 7

Trigger the Test Writer

Now we ask for tests. Claude delegates to the test-writer agent, which creates and runs tests in its own worktree.

~/my-project — claude
> Write comprehensive unit tests for the users module

▶ Delegating to agent: test-writer
model: sonnet • isolation: worktree • tools: Read, Grep, Glob, Edit, Write, Bash

Creating worktree at /tmp/claude-wt-7b2e1f...
Agent started • analysing source...

│ Reading src/users.js...
│ Writing tests/users.test.js...
│ Running: npm test...
│ 1 test failed — fixing mock setup...
│ Editing tests/users.test.js...
│ Running: npm test...
│ All 7 tests passing

✓ Agent complete (18 tool calls, 1m 12s)

Created tests/users.test.js with 7 tests:
• GET /users/:id — returns user for valid ID
• GET /users/:id — returns 404 for missing user
• GET /users/:id — handles non-numeric ID
• POST /users — creates user with valid data
• POST /users — rejects missing required fields
• POST /users — rejects duplicate email
• POST /users — handles database errors

Worktree branch: claude/test-writer-7b2e1f
Changes preserved — merge with: git merge claude/test-writer-7b2e1f

Self-healing The agent's first test run failed, so it read the error, fixed the mock setup, and re-ran — all within its 30-turn budget. The worktree is preserved because the agent made changes.

STEP 8

Merge & Verify

Back in the main session, we merge the test-writer's branch and run the full suite.

~/my-project — claude
> Merge the test-writer's changes and fix the SQL
injection the reviewer found

│ Running: git merge claude/test-writer-7b2e1f
│ Merge successful
│ Editing src/users.js:14 — parameterised query
│ Editing src/users.js:28 — adding email validation
│ Running: npm test

PASS tests/users.test.js

✓ GET /users/:id — returns 404 for missing user (2ms)
✓ GET /users/:id — handles non-numeric ID (1ms)
✓ POST /users — creates user with valid data (3ms)

Complete loop Code-reviewer found the bugs → test-writer created the safety net → main session applied both. Each agent stayed in its lane with its own tool set and isolation boundary.

SUMMARY

What We Covered

Agent Definition

YAML frontmatter in .claude/agents/<name>/AGENT.md controls tools, model, isolation, and turn limits

Auto-Delegation

Claude matches task descriptions to agent descriptions — no explicit routing needed

Worktree Isolation

Each agent gets its own Git worktree — changes are preserved on a branch if modifications were made

Self-Healing Loops

Agents can run tests, read errors, fix code, and re-run — all within their turn budget

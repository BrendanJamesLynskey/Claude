Claude Code Desktop

The Native GUI for Agentic Coding

The same agent as the CLI, wrapped in a native desktop app — sessions, diffs, permissions, and MCP servers in a single window. For developers who want Claude Code without living in the terminal.

Desktop App Agentic macOS · Windows · Linux Same Core Engine
01

What Is Claude Code Desktop?

Claude Code Desktop is a native application that wraps the Claude Code agent in a graphical interface. It is not a different agent, a different model, or a different feature set — it is the same Claude Code you know from the CLI, with a GUI layered on top for users who prefer visual tools.

Same Core, New Surface

  • Identical underlying agent to the CLI
  • Same tools (Read, Edit, Bash, Grep, Glob)
  • Same hooks, skills, sub-agents
  • Same MCP server ecosystem

What’s New

  • Native windowing & multi-session layout
  • Visual diff viewer for every edit
  • Permission prompts as OS-native dialogs
  • Project picker and workspace switcher

What It Is Not

  • Not an IDE — it doesn’t edit files by hand
  • Not Claude.ai — it runs locally
  • Not a replacement for the CLI — both coexist
  • Not limited in capability vs. the terminal

Positioning: Claude Code Desktop targets developers who want the agent but find the terminal UX limiting — PMs, designers, data scientists, and developers who prefer a GUI for reviewing changes and managing sessions.

02

Why a Desktop App?

Limits of the Terminal UX

  • Diff review is painful with wrapped lines and tiny fonts
  • Managing multiple parallel sessions requires tmux or several terminal windows
  • Permission prompts interrupt the typing flow
  • Copy/paste across sessions is clumsy
  • Not every developer is a terminal power user

What the GUI Unlocks

  • Side-by-side diff view with syntax highlighting
  • Multiple sessions in tabs or panes
  • OS-native notifications on long-running tasks
  • Searchable session history and transcripts

Design Principle

The desktop app never hides the underlying agent. Every action is still a tool call, every edit is still reviewable, every permission is still explicit. The GUI surfaces the agent’s work — it does not obscure it.

Same Skill Tree

Skills, slash commands, hooks, CLAUDE.md, and settings.json all work identically. Users who learn in the CLI can switch to desktop without relearning.

Not an IDE Replacement

Users still open their editor of choice for manual editing. The desktop app is an agent cockpit, not a code editor.

First Launch & Authentication

Onboarding Flow

  1. Welcome screen with product tour
  2. Authentication choice:
    • Claude.ai subscription — OAuth in browser
    • API key — console.anthropic.com
    • Enterprise SSO — SAML/OIDC
  3. Permission grants (filesystem, notifications, accessibility)
  4. Pick an initial workspace (optional)
  5. Drop into an empty session ready for input

Credential Storage

  • macOS → Keychain
  • Windows → Credential Manager
  • Linux → libsecret (gnome-keyring, kwallet)
  • Never stored in plaintext config

Billing Model

  • Included with Claude Pro / Max / Team / Enterprise
  • Usage counts against the same subscription quota as Claude Code CLI
  • Can also attach an API key — billed per-token to the Anthropic Console account
  • Mixed mode: some workspaces on subscription, others on API key

Privacy Defaults

  • Telemetry opt-in only
  • No code sent to Anthropic beyond the API payload you authorise
  • Local session storage, encrypted at rest
  • You can wipe all local data from Settings → Privacy

The Main Interface

Three-pane layout: sidebar for workspaces and tools, transcript in the middle, inspector on the right for live diffs, status, and actions.

06

Workspaces & Projects

Workspaces

A workspace is a named directory that Claude Code treats as a project root. Each workspace has its own CLAUDE.md, settings.json, MCP configuration, and session history.

  • Drag-and-drop a folder to create a workspace
  • Or File → Open Workspace…
  • Switch workspaces from the sidebar dropdown
  • Multiple workspaces can be pinned simultaneously

Per-Workspace Settings

  • Tool allow/deny lists persist per workspace
  • MCP servers scoped to the workspace
  • Custom slash commands per project
  • Shared via .claude/ checked into git

Worktree-Aware

When you open a directory that contains git worktrees, the desktop app surfaces them as sibling workspaces — same repo, different branches, independent sessions. Lets you run multiple agents in parallel on the same codebase.

Session History

Every session is saved locally with full transcript, tool calls, diffs, and final state. Searchable across workspaces. Exportable as JSON or Markdown.

Resuming Sessions

Reopen a previous session and the desktop app rebuilds the full context (files read, tools available, prior messages) so you can pick up where you left off. Equivalent to the CLI’s --resume flag.

Multi-Session

Split view lets you run two sessions side-by-side in the same workspace. Useful for comparing approaches or running a long task in one pane while exploring in the other.

07

Diff Viewer & Change Review

Every file edit appears in the Inspector panel as a side-by-side diff with syntax highlighting. This is one of the biggest wins over the CLI — reviewing changes no longer requires squinting at a wall of +/- lines in a terminal.

Review Actions

  • Accept — commit the edit to disk
  • Revert — undo the edit
  • Refine — ask Claude to modify the proposal
  • Open in Editor — hand off to VS Code, Cursor, etc.

Batch Review

  • When Claude modifies multiple files, review them one at a time
  • Skip forward and back with arrow keys
  • Accept all, accept changed-subset, or revert all

Diff Modes

ModeUse Case
Side-by-sideDefault for wide screens
InlineNarrow windows or long lines
Word-levelFine-grained prose changes
Hunks onlyFocus on what changed

Syntax Highlighting

Monaco-based editor (same engine as VS Code). Language detection from file extension; 100+ languages supported out of the box.

Git Integration

If the workspace is a git repo, the diff viewer also shows how each edit relates to the current branch HEAD. You can stage hunks, generate commit messages, and open PRs without leaving the app.

08

Plan Mode & Permission Controls

Plan Mode

The PLAN button in the header toggles a read-only mode in which Claude can investigate but cannot modify anything. Useful for surveying a codebase, drafting an approach, or asking “what would you do?” without any risk.

  • All read tools work: Read, Grep, Glob, Bash (read-only subset)
  • Write tools are disabled
  • Exit plan mode to execute the plan

Permission Prompts

Permission decisions are OS-native dialogs rather than terminal prompts. Three levels per tool:

  • Allow once — one-shot approval
  • Allow always — persist for this session
  • Deny — block and explain why

Permission Modes

ModePrompts
DefaultAsk for each dangerous tool
PlanNo writes at all
TrustedAuto-allow pre-approved tools
YoloAll tools auto-allowed — for isolated envs only

Danger Warnings

The desktop app shows red banners for destructive operations: rm -rf, git push --force, DROP TABLE, and similar. You must explicitly confirm, even in Trusted mode.

Rules Engine

The same settings.json permission rules as the CLI. Block specific commands, allow-list safe ones, require confirmation on writes outside the workspace.

09

MCP Servers in the Desktop App

The Model Context Protocol ecosystem works identically to the CLI — the desktop app is a full MCP host. What’s new is the GUI around server management.

MCP Server Panel

  • List all configured servers with health indicators
  • One-click enable / disable per workspace
  • Browse each server’s tools and resources
  • Live logs when debugging a misbehaving server
  • Add server from a URL, a command, or the MCP Registry

Transport Support

  • stdio (local processes)
  • Streamable HTTP (remote servers)
  • SSE (legacy, for compatibility)

Registry Integration

Browse and install MCP servers from the public registry without editing JSON config files. Search by capability (“database,” “github,” “jira”), preview the tool list, and click to install.

OAuth Flows

Remote MCP servers requiring OAuth 2.1 open the browser for user consent. Tokens are stored in the OS keychain and refreshed automatically. No manual token copying.

Sandboxed Local Servers

Local stdio MCP servers run with the same sandboxing as the agent itself. Each server gets an isolated working directory and cannot see other workspaces.

Config Portability

MCP configuration lives in .mcp.json — same as the CLI. A project’s MCP setup works identically whether you open it in the desktop app or the terminal.

Sub-agents, Skills, and Hooks

Sub-agents

  • Same sub-agent definitions as the CLI
  • Managed from the sidebar
  • Visual graph of parent/child agent calls
  • Drill into any sub-agent’s transcript independently
  • Background sub-agents stream progress to notifications

Skills

  • Skills library UI with descriptions and tags
  • Enable / disable per workspace or globally
  • Install from marketplace or local directory
  • Skill author mode with live reload
  • See which skills were used in each session

Hooks

  • Visual hook manager in settings
  • Shows current PreToolUse, PostToolUse, Stop, etc.
  • Test hook firing from the UI
  • Inspect captured inputs and outputs per hook
  • Same shell-command contract as the CLI

Key principle: all of these feature surfaces live in the same files as the CLI (~/.claude/agents/, .claude/skills/, settings.json). The desktop app is a view onto your Claude Code configuration — it never introduces its own parallel config system.

12

Desktop ↔ CLI ↔ Cloud Interop

One configuration, three surfaces. Desktop, CLI, and Cloud all read the same CLAUDE.md, the same settings.json, the same .mcp.json, and the same sub-agent definitions. You can start a session in the desktop app, resume it in the CLI on an SSH box, and hand off to a cloud agent for a long-running task — all without reconfiguration.

When to Use Which Surface

Scenario Recommended surface Why
Daily feature work on a local project Desktop or IDE extension Visual diff review, fast iteration, persistent sessions
Scripting, CI/CD, headless automation CLI No GUI, composable with shell, runs unattended
Remote dev box over SSH CLI No display needed; works over any terminal
Reviewing an agent’s large refactor Desktop Side-by-side diff viewer and accept/revert per file
Pair-programming inside VS Code / JetBrains IDE extension Inline code actions and editor context
PR review on GitHub, issue triage Cloud (GitHub App) Runs where the work lives; no local setup
Long-running background task (hours) Cloud or CLI in tmux Survives laptop sleep; notifications on completion
Onboarding a non-terminal user Desktop Lowest friction, GUI-native onboarding
Managing many parallel sessions Desktop Tabs, split panes, notifications
Reviewing an MCP server’s tool surface Desktop Visual tool explorer and log panel

Practical pattern: use the desktop app as your daily driver, drop to the CLI for automation and remote boxes, and use the cloud agents for PR workflows. All three share the same config, so there is no cost to mixing and matching.

14

Limitations & Closing Notes

Known Limitations

  • No headless mode — the desktop app requires a window; use the CLI for automation
  • Workspace must be local — no direct remote filesystem support (use CLI over SSH instead)
  • Heavier startup than the CLI (Electron-based bundle)
  • Updates lag the CLI by a few days for new features
  • Accessibility prompts on macOS require Accessibility and Full Disk Access for full filesystem tools

Not Included (Yet)

  • Built-in code editor (intentional — not an IDE)
  • Team session sharing (on the roadmap)
  • Native mobile companion

When to Avoid It

  • You live in tmux and your workflow is terminal-first
  • You primarily work on remote dev boxes over SSH
  • Your use case is unattended automation
  • You need lowest-possible memory footprint

When It Shines

  • You want to review diffs visually before accepting
  • You manage multiple projects / worktrees in parallel
  • You collaborate with non-terminal users
  • You care about polished permission dialogs and notifications
  • You want to browse MCP servers and skills visually

The Big Picture

Claude Code Desktop is not a different product — it is a new surface for the same agent. The winning pattern is to mix surfaces based on the task, not to pick one and stick with it. One configuration, many cockpits.
